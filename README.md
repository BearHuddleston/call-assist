# Call Assist

Call Assist is a supervised, text-first calling copilot for Deaf and hard-of-hearing people. The user defines a low-risk goal, facts the assistant may share, and hard boundaries; Call Assist creates a reviewable plan, conducts the conversation with large two-speaker captions, and stops for approval before any commitment.

This project is being built for OpenAI Build Week in the **Apps for Your Life** category. “Call Assist” is a working title.

## What works now

- Accessible setup form with allowlisted demo destinations, facts, boundaries, and a low-risk confirmation
- Server-side safety screening for emergency, payment, high-stakes healthcare/financial, and outreach requests
- Reviewable call plans generated through the OpenAI Responses API with GPT-5.6 Sol when configured
- Fully local judge-safe fallback when no API key is present
- Supervised call simulation with large two-speaker captions
- Typed “say this,” correction, pause/resume, explicit approval, decline, and hang-up controls
- AI/accessibility disclosure and affirmative transcription consent in the opening exchange
- Structured post-call outcome generated by GPT-5.6 when configured
- No audio recording and transcript deletion after the outcome is created
- Credential-gated Twilio outbound calling service with a bidirectional Media Stream bridge to OpenAI Realtime
- Normalized call-event and supervisor-command proxy routes that keep OpenAI and Twilio credentials off the browser
- Deliberate live-call mode with a custom allowlisted destination, visual ringing/live/paused states, cursor-based captions, and real approval/control commands
- Automatic transition from a completed or failed live call into the same structured, transcript-cleared outcome flow

The browser experience remains in judge-safe simulation mode by default. Live mode appears only when the private calling service reports that OpenAI, Twilio, an HTTPS webhook endpoint, and the service-to-service token are configured. A real destination must still pass the server-side allowlist before a call starts.

## Run locally

Requirements: Node.js 22.13 or newer.

```bash
npm install
cp .env.example .env.local
npm run dev
```

Open the local URL printed by the development server. Demo mode is on by default and does not need credentials.

To exercise real GPT-5.6 planning and outcomes, set `CALL_ASSIST_DEMO_MODE=false` and add `OPENAI_API_KEY` to `.env.local`. API keys are only read on the server.

## Run the live calling service

Fill the telephony values in `.env.local`, including a long random `CALL_ASSIST_SERVICE_TOKEN`, an HTTPS `TELEPHONY_PUBLIC_BASE_URL`, and a real E.164 test number in `CALL_ASSIST_ALLOWLIST`. Then run both processes:

```bash
npm run dev
npm run telephony:dev
```

The separate Fastify service owns the long-lived Twilio Media Stream and OpenAI Realtime session. The web app reaches it only through authenticated server-side routes:

- `GET /api/live/status`
- `POST /api/live/start`
- `GET /api/live/:callId/events?after=0`
- `POST /api/live/:callId/commands`

Keep `TWILIO_VALIDATE_SIGNATURES=true` outside a local webhook harness. Do not place a live call until the destination is explicitly allowlisted and the person operating the demo is ready to supervise it.

Once both processes are healthy, choose **Allowlisted live destination…** in the setup screen. The UI will collect the business name and E.164 number, create a reviewable plan, and offer both the safe simulation and supervised live-call buttons.

See [docs/live-call-runbook.md](docs/live-call-runbook.md) before the first controlled call.

## Three-minute demo path

1. Use the prefilled library-room request and confirm it is a low-risk call.
2. Create the call plan and point out the disclosure, success criteria, and approval gates.
3. Start the supervised demo call and show the large two-speaker captions.
4. Pause the assistant, type a message, or enter a correction.
5. Approve the room reservation when Call Assist stops.
6. Show the structured outcome, confirmation number, and transcript-cleared privacy state.

For a live demo, repeat the same path with one consented, allowlisted test destination. Keep the safe simulation ready as the submission fallback.

## Current architecture

- **Web app:** React 19 and Next-compatible App Router compiled with vinext
- **Planning and outcomes:** OpenAI Responses API, `gpt-5.6-sol`, strict Zod-backed structured outputs
- **Voice service:** OpenAI Realtime API, `gpt-realtime-2.1`, through the Agents SDK Twilio transport
- **Telephony service:** Fastify, Twilio outbound calls, and bidirectional Media Streams
- **Hosting target:** Cloudflare Worker-compatible output through OpenAI Sites

The judge-safe web application can run on Sites. The Fastify telephony process needs separate Node hosting with a public HTTPS/WSS endpoint because it owns long-lived provider WebSockets.

## Safety scope

- User-initiated and allowlisted calls only
- Low-risk direct destinations; no dynamic IVR or DTMF in the MVP
- Accurate AI/accessibility disclosure and affirmative consent before live transcription
- No emergency calls, telemarketing, bulk outreach, payments, or high-stakes healthcare/financial transactions
- No audio recording; ephemeral captions by default
- Prominent pause, correction, approval, decline, and end-call controls
